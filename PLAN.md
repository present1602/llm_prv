# 특허 명세서 초안 자동 작성 — 폐쇄형 LLM 구축 플랜

## 1. 목표

발명자 또는 변리사가 입력한 **발명 설명(기술 요지, 구성, 효과, 도면 설명 등)** 을 바탕으로,
한국 특허청(KIPO) 서식에 맞는 **명세서 초안 전체를 자동 생성**하는 시스템을 구축한다.

- **폐쇄형(On-premise/Closed)**: 외부 API(OpenAI, Claude 등)를 사용하지 않고,
  사내 서버(또는 로컬 장비)에서 오픈소스 LLM을 직접 호스팅한다.
  → 발명 내용은 출원 전 비밀유지가 생명이므로 데이터가 외부로 나가지 않는 것이 핵심 요구사항.

### 생성 대상 (특허법 시행규칙 명세서 서식 기준)

| 섹션 | 비고 |
|---|---|
| 【발명의 명칭】 | 국문 + 영문 |
| 【기술분야】 | |
| 【발명의 배경이 되는 기술】 | 선행기술 인용 (RAG 연계 대상) |
| 【해결하고자 하는 과제】 | |
| 【과제의 해결 수단】 | 청구항과 정합성 필요 |
| 【발명의 효과】 | |
| 【도면의 간단한 설명】 | |
| 【발명을 실시하기 위한 구체적인 내용】 | 가장 분량이 크고 중요 |
| 【청구범위】 (독립항/종속항) | 품질의 핵심. 별도 생성 단계로 분리 |
| 【요약서】 + 【대표도】 | |

## 2. 핵심 설계 원칙

1. **섹션별 파이프라인 생성**: 명세서를 한 번에 생성하지 않고,
   `청구항 설계 → 과제/수단/효과 → 상세한 설명 → 배경기술 → 명칭/요약` 순으로
   단계별 생성하며 앞 단계 결과를 뒤 단계의 컨텍스트로 주입한다.
   (청구항 ↔ 상세설명 간 용어·도면부호 일관성 확보가 목적)
2. **모델은 교체 가능하게**: 서빙 레이어(OpenAI 호환 API)를 추상화하여
   모델 업그레이드(7B → 32B → 파인튜닝 모델)가 코드 수정 없이 가능하도록 한다.
3. **초안(draft) 포지셔닝**: 최종 산출물은 "변리사가 검토·수정하는 초안"이며,
   사람 검토를 대체하지 않는다. UI도 편집 가능한 에디터 중심으로 설계.

## 3. 기술 스택 (제안)

### 3.1 LLM (폐쇄망 자체 호스팅)

| 후보 모델 | 특징 |
|---|---|
| **EXAONE 3.5 / 4.0 (LG AI연구원)** | 한국어 최상급, 7.8B/32B, 상업 라이선스 확인 필요 |
| **Qwen2.5 / Qwen3 (7B~72B)** | 한국어 우수, Apache 2.0, 긴 컨텍스트(128K) |
| Llama 3.x, Gemma 3 | 대안 |

- **MVP**: Ollama + 7~14B 모델 (Mac/단일 GPU에서 구동, 빠른 검증)
- **운영**: vLLM + 32B급 (양자화) — GPU 서버 (A6000/L40S/A100 급 1~2장)
- 명세서는 길기 때문에 **긴 컨텍스트(32K 이상)** 지원 모델 필수

### 3.2 시스템 구성

```
[웹 UI]  ──  [백엔드 API (FastAPI)]  ──  [LLM 서빙 (Ollama/vLLM, OpenAI 호환)]
                    │
                    ├── [생성 파이프라인 오케스트레이터] (섹션별 프롬프트 체인)
                    ├── [벡터 DB (Qdrant/Chroma)] ← 선행특허·기존 명세서 RAG (Phase 2)
                    └── [문서 출력기] (Markdown → DOCX/HWP 변환)
```

- 백엔드: Python + FastAPI
- 프론트: React (또는 MVP는 단일 페이지 HTML) — 입력 폼 + 섹션별 초안 에디터
- 출력: Markdown 초안 → `python-docx`로 DOCX 변환 (HWP는 Phase 4)

### 3.3 데이터 (Phase 2~3)

- **KIPRIS / 특허청 공개 데이터**: 공개 특허공보 명세서 corpus (RAG 및 파인튜닝 학습용)
- 사내 보유 명세서 (품질 좋은 few-shot 예시 및 파인튜닝 데이터)

## 4. 단계별 로드맵

### Phase 1 — MVP (지금 개발할 것)

**목표**: "발명 설명 입력 → KIPO 서식 명세서 초안 출력" 엔드투엔드 검증

- [ ] Ollama 설치 + 한국어 모델 선정 (EXAONE 3.5 7.8B vs Qwen2.5 14B 비교 테스트)
- [ ] 발명 입력 스키마 정의 (발명 명칭 가안, 기술분야, 종래기술 문제점, 핵심 구성, 효과, 도면 설명, 실시예 메모)
- [ ] 섹션별 생성 파이프라인 구현 (프롬프트 체인, 청구항 우선 생성)
- [ ] 웹 UI: 입력 폼 → 생성 진행상황 표시 → 섹션별 초안 확인/편집
- [ ] Markdown / DOCX 다운로드
- **범위 제외**: RAG, 파인튜닝, HWP, 사용자 인증

### Phase 2 — RAG (선행기술·스타일 참조)

- KIPRIS 공보 데이터 수집 → 임베딩(BGE-M3 등 로컬 임베딩 모델) → 벡터 DB 구축
- 배경기술 섹션에 유사 선행특허 자동 인용
- 동일 기술분야의 우수 명세서를 few-shot 스타일 참조로 주입

### Phase 3 — 도메인 특화 (파인튜닝 + 평가)

- 명세서 corpus 기반 LoRA/QLoRA 파인튜닝 (입력요지 → 섹션 생성 쌍 데이터셋 구축)
- 평가 체계: 서식 준수율, 청구항-상세설명 용어 정합성, 변리사 블라인드 평가

### Phase 4 — 운영 품질

- 자동 검증기: 서식 검사, 도면부호 일관성, 청구항 인용관계 검사
- HWP 출력, 특허로 제출 서식 연계
- 폐쇄망 배포 패키징(Docker), 접근제어/감사 로그

## 5. 리스크 및 대응

| 리스크 | 대응 |
|---|---|
| 소형 모델의 법률문체·청구항 품질 한계 | 섹션 분할 + 상세 프롬프트로 보완, Phase 3 파인튜닝, 필요시 32B급 승급 |
| 환각(존재하지 않는 선행기술 인용) | MVP에서는 선행기술 인용 생성 금지 → Phase 2 RAG로만 인용 |
| GPU 인프라 비용 | MVP는 Ollama+양자화 모델로 저비용 검증 후 투자 판단 |
| 모델 라이선스 (상업적 사용) | 모델 선정 시 라이선스 검토 (Qwen: Apache 2.0 안전) |

## 6. MVP 산출물 정의

- `backend/`: FastAPI 서버 (생성 파이프라인 + Ollama 연동)
- `frontend/`: 입력 폼 + 초안 에디터 웹 UI
- 명세서 초안: 화면 확인 + Markdown/DOCX 다운로드
- 성공 기준: 발명 요지 1건 입력 시 5분 이내에 전 섹션이 채워진 서식 준수 초안 생성
