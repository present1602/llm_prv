# Phase 2 상세 플랜 — RAG (선행기술·유사특허 연동)

> 상위 플랜: [DRAFT_PLAN.md](../DRAFT_PLAN.md) · [PLAN.md](PLAN.md)
> 본 문서는 Phase 2의 실행 단위 플랜(v1)이다. Phase 1 MVP는 완료됨을 전제로 한다.

---

## 0. 목표와 배경

### 왜 RAG인가
Phase 1 MVP는 **발명자 입력 범위 내에서만** 명세서를 생성한다(환각 억제).
그 결과 다음이 부족하다.

- **배경기술** 섹션이 빈약함 — 실제 종래기술·선행문헌 근거가 없음
- **청구범위**의 권리범위 설정 시 유사특허와의 차별점을 반영 못 함
- 명세서 **문체·형식**이 실제 등록 특허의 관례를 따르지 못함

Phase 2는 **KIPRIS 등 공개 특허 corpus를 검색(RAG)** 하여 위 세 지점에 근거·예시를
주입한다. 핵심 원칙은 유지: *생성물은 검색된 실제 문서에 근거하며, 없는 것을 지어내지 않는다.*

### Phase 2 완료 정의 (Definition of Done)
1. 발명 설명 입력 시, 관련 공개특허 상위 N건을 검색해 화면에 근거로 표시한다.
2. 배경기술·청구범위 생성이 검색된 선행문헌을 인용/참고해 작성된다.
3. 유사 명세서의 동일 섹션을 few-shot 예시로 자동 선별해 프롬프트에 주입한다.
4. 모든 검색·임베딩·저장은 **로컬(폐쇄형)** 에서 수행된다(외부 API 미사용).

## 1. 범위

### 포함
- 공개 특허 데이터 수집·정규화 파이프라인
- 로컬 임베딩 + 벡터 DB 구축·색인
- 검색(retrieval) API 및 리랭킹
- 생성 파이프라인에 컨텍스트 주입(배경기술·청구범위·few-shot)
- 프런트: 근거(출처) 표시 UI
- 검색·생성 품질 평가

### 제외 (Phase 3+로 이연)
- 모델 파인튜닝(LoRA 등) — Phase 3
- HWP 출력, 도면 인식 — Phase 4
- 사용자 인증·과금 — Phase 4

## 2. 아키텍처 (Phase 1 대비 추가분)

```
[웹 UI] ──► [백엔드 API (FastAPI)]
                 │
                 ├─ (기존) 스키마·프롬프트 오케스트레이터
                 ├─ (신규) Retriever ─────────► [벡터 DB (Qdrant)]
                 │            │                      ▲
                 │            └─ 임베딩(BGE-M3, 로컬) │
                 │                                   │
                 └─ (기존) LLM 서버(Ollama)     [색인 파이프라인]
                                                     ▲
                                          [수집·파싱·청킹] ← KIPRIS/공개 특허
```

- **벡터 DB**: Qdrant (로컬 Docker). 메타데이터 필터(IPC 분류, 출원일 등) 지원이 좋아 Chroma보다 선호.
- **임베딩**: BGE-M3 (다국어·한국어 양호, 로컬 구동). 대안: KURE, ko-sbert.
- **리랭커**: bge-reranker-v2-m3 (선택, 정밀도 향상용).

## 3. 작업 분해 (스텝별)

### Step 1 — 데이터 수집 (Corpus)
- [ ] KIPRIS Plus OpenAPI 신청·키 발급 (또는 공개 특허 공보 벌크 데이터 확보)
- [ ] 수집 대상 정의: 우선 **1~2개 기술분야(IPC)** 로 좁혀 파일럿 (예: 물리/전기 일부)
- [ ] 수집 스크립트: 서지정보 + 명세서 전문(배경기술·청구범위 포함) 저장
- [ ] 저장 포맷: 문서당 JSON (출원번호, 명칭, IPC, 출원일, 섹션별 본문, 원문 링크)
- [ ] 목표 규모(파일럿): **5,000~20,000건** (품질 검증엔 충분, 나중에 확장)
- 리스크: KIPRIS 이용약관/저작권 확인. 상업 이용 조건·저장 허용 범위 사전 검토.

### Step 2 — 파싱·정규화·청킹
- [ ] 명세서 섹션 분리(배경기술/해결수단/청구범위/실시예 등) — 기존 `schema.py` 항목과 매핑
- [ ] 텍스트 정제(도면부호, 표, 각주 처리)
- [ ] 청킹 전략: **섹션 단위 + 길이 제한**(예: 512~1024 토큰, 오버랩 128)
- [ ] 청크 메타데이터: doc_id, section_key, ipc, 출원일

### Step 3 — 임베딩·색인
- [ ] BGE-M3 로컬 로딩(sentence-transformers 또는 FlagEmbedding)
- [ ] Qdrant 컬렉션 설계: vector(dense) + payload(메타) + (선택)sparse 하이브리드
- [ ] 배치 임베딩·업서트 파이프라인 (재실행 가능·증분 색인)
- [ ] 색인 검증: 샘플 쿼리 → 관련 문서 상위 노출 확인

### Step 4 — Retriever API
- [ ] `POST /api/retrieve` : 발명 설명 → 관련 특허 top-k (메타 필터 옵션)
- [ ] 하이브리드 검색(dense + BM25/sparse) → 리랭킹(선택)
- [ ] 반환: 문서 스니펫 + 출처(출원번호·명칭·링크) + 유사도 점수
- [ ] 백엔드 설정: `EMBED_MODEL`, `QDRANT_URL` 환경변수로 교체 가능하게

### Step 5 — 생성 파이프라인 통합
- [ ] `prompts.py` 확장: 배경기술·청구범위 프롬프트에 **검색 컨텍스트 블록** 주입
- [ ] few-shot 자동 선별: 생성할 섹션과 동일 section_key의 유사 청크를 예시로 첨부
- [ ] 근거 강제: "제공된 선행문헌에 없는 특정 문헌번호/인용은 생성 금지" 규칙 유지·강화
- [ ] `generate`/`regenerate` 요청에 `use_rag` 플래그 추가(온/오프 비교 가능)

### Step 6 — 프런트 (근거 표시)
- [ ] 생성 결과 옆에 **참고한 선행문헌 목록**(출처·링크·유사도) 패널
- [ ] 배경기술·청구범위 섹션에 근거 배지/인용 표시
- [ ] RAG on/off 토글 (품질 체감 비교용)

### Step 7 — 평가
- [ ] **검색 품질**: 소규모 정답셋으로 Recall@k, MRR 측정
- [ ] **생성 품질**: RAG on/off A/B — 배경기술 근거성, 청구범위 차별성, 변리사 체감 수정량
- [ ] 환각 점검: 생성된 인용/문헌번호가 실제 검색 결과에 존재하는지 자동 검증
- [ ] 지연시간: 검색+생성 총 소요 시간(목표: 섹션당 허용 범위 내)

## 4. 신규 컴포넌트 (코드 배치안)

```
backend/app/
├── rag/
│   ├── ingest.py       # 수집→파싱→청킹 (Step 1~2)
│   ├── embed.py        # BGE-M3 임베딩 래퍼 (Step 3)
│   ├── store.py        # Qdrant 클라이언트·색인·검색 (Step 3~4)
│   └── retriever.py    # 하이브리드 검색+리랭킹, few-shot 선별 (Step 4~5)
├── prompts.py          # (수정) 검색 컨텍스트 주입
└── main.py             # (수정) /api/retrieve, use_rag 플래그
scripts/
└── build_index.py      # 오프라인 색인 실행 진입점
docker-compose.yml      # Qdrant 로컬 구동
```

## 5. 기술 선택 근거 (요약)

| 항목 | 선택 | 이유 | 대안 |
|---|---|---|---|
| 벡터 DB | Qdrant | 메타 필터·하이브리드·로컬 Docker 용이 | Chroma, Weaviate |
| 임베딩 | BGE-M3 | 한국어·다국어 양호, 로컬, 롱 청크 | KURE, ko-sbert |
| 리랭커 | bge-reranker-v2-m3 | 정밀도↑ (선택 적용) | 미적용(속도 우선) |
| 청킹 | 섹션+길이 | 특허 구조 반영, 섹션별 few-shot 용이 | 고정 길이 |

## 6. 리스크 & 대응

| 리스크 | 대응 |
|---|---|
| KIPRIS 데이터 이용약관·저작권 | 수집 전 약관 검토, 파일럿은 소규모·내부 검증용으로 한정 |
| 임베딩/검색이 한국어 특허 용어에 약함 | 정답셋으로 조기 평가, 필요 시 리랭커·하이브리드 강화 |
| 검색 지연으로 UX 저하 | 색인 사전 구축(오프라인), top-k 축소, 캐싱 |
| RAG가 오히려 환각 유발(잘못된 인용) | 검색 결과 존재 검증 자동화, 없는 인용 생성 금지 규칙 |
| 데이터 규모 확장 시 비용·시간 | 파일럿(IPC 한정)으로 검증 후 점진 확장 |

---

## 7. 마일스톤 (제안)

1. **M1 — 파일럿 인덱스**: IPC 1~2개, 5k건 색인 + `/api/retrieve` 동작 (Step 1~4)
2. **M2 — RAG 생성**: 배경기술·청구범위에 컨텍스트 주입, 근거 표시 UI (Step 5~6)
3. **M3 — 평가·튜닝**: A/B 평가, 리랭커·청킹 튜닝, 환각 검증 (Step 7)
4. **M4 — 확장 판단**: corpus 확대 여부·Phase 3(파인튜닝) 진입 결정
