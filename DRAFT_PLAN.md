# 특허 명세서 초안 자동 작성 — 폐쇄형(Closed) LLM 구축 플랜

> 목적: 발명자의 아이디어 입력(발명 설명, 도면, 배경기술 등)을 받아,
> 한국 특허청(KIPO) 양식에 맞는 **특허 명세서 초안**을 자동 생성하는
> **폐쇄형(On-premise / 외부망 미연동) LLM 시스템**을 구축한다.
> 본 문서는 상위 플랜이며, 이를 기반으로 MVP를 개발한다.

---

## 1. 왜 폐쇄형(Closed)인가

- **기밀성**: 미공개 발명 내용은 최고 수준의 기밀. 출원 전 아이디어가 외부
  API(OpenAI, Claude 등)로 유출되면 신규성 상실·특허성 훼손 위험이 있다.
- **규제·계약**: 고객(기업/개인 발명자)과의 비밀유지 계약(NDA), 특허사무소의
  보안 요건상 데이터가 사내망을 벗어나면 안 되는 경우가 많다.
- **결론**: 오픈소스 LLM(가중치 다운로드 가능)을 자체 인프라(온프레미스/사설
  클라우드)에서 구동한다. 외부로 나가는 네트워크 호출은 원칙적으로 차단.

---

## 2. 핵심 설계 원칙

1. **구조화된 입력 → 구조화된 출력**: 자유서술 발명 설명을 그대로 던지지 않고,
   `발명의 명칭 / 기술분야 / 배경기술 / 해결과제 / 해결수단 / 효과 / 실시예 / 청구항`
   같은 KIPO 명세서 항목(스키마)에 맞춰 단계적으로 채운다.
   → 환각(hallucination)을 줄이고, 각 섹션을 독립적으로 재생성·수정 가능.
2. **로컬 우선(Local-first)**: MVP는 Ollama + 중형 모델(7~14B)로 노트북/단일 GPU에서
   동작. 이후 vLLM + 대형 모델(32B~)로 품질·처리량 확장.
3. **초안(draft) 도구**: "완성본"이 아니라 **변리사·발명자가 검토·수정하는 초안**을
   빠르게 만드는 도구. 사람이 최종 책임. UI에서 섹션별 재생성/편집 지원.
4. **문서 파싱 기반**: 발명자가 제출하는 기존 자료(출원서, 도면 설명, 선행문헌 PDF 등)를
   파싱해 입력 컨텍스트로 활용 (`file_samples/`의 실제 KIPO 문서가 참조 자료).

---

## 3. 기술 스택 (제안)

### 3.1 LLM (폐쇄형 오픈소스)

| 후보 모델 | 비고 |
|---|---|
| **EXAONE 3.5 / 4.0 (LG AI연구원)** | 한국어 특화, 7.8B/32B, 국내 문서 처리 유리 |
| **Qwen2.5 / Qwen3 (7B~72B)** | 한국어 준수, Apache 2.0, 롱 컨텍스트(128K) |
| Llama 3.x, Gemma 3 | 대안, 라이선스·한국어 성능 비교 필요 |

- **MVP**: Ollama + 7~14B (Mac/단일 GPU에서 로컬 구동, 비용 최소)
- **확장**: vLLM + 32B급 → GPU 서버(A6000/L40S/A100 1~2장), 처리량↑ 지연↓
- 명세서는 문서가 길어 **롱 컨텍스트(32K 이상)** 지원 모델 우선

### 3.2 시스템 구성

```
[웹 UI]  ──►  [백엔드 API (FastAPI)]  ──►  [LLM 서버 (Ollama/vLLM, OpenAI 호환)]
                    │
                    ├─ [문서 파서 (PDF/HWP → 텍스트)]   ← file_samples/ 활용
                    ├─ [명세서 스키마·프롬프트 오케스트레이터]
                    ├─ [벡터 DB (Qdrant/Chroma)] ← 선행문헌·유사특허 RAG (Phase 2)
                    └─ [문서 생성기] (Markdown → DOCX / HWP)
```

- 백엔드: Python + FastAPI. **LLM 호출은 OpenAI 호환 클라이언트로 추상화**하여
  MVP는 Ollama, 이후 vLLM으로 교체 가능하게 유지 *(확정)*
- 프론트: **React 앱** — 입력 폼 + 섹션별 결과 편집·재생성 UI *(확정)*
- 파싱: `pdfplumber` / `PyMuPDF`(PDF), HWP는 Phase 4
- 출력: Markdown 우선 → `python-docx`로 DOCX 변환 (HWP는 Phase 4)

### 3.3 데이터

- `file_samples/` : 실제 KIPO 출원서·등록증·특허로 문서 (양식·구조 참조, few-shot 예시)
- KIPRIS / 공개 특허 데이터: 선행기술 corpus (RAG용, Phase 2)

## 4. 로드맵

### Phase 1 — MVP (본 개발 대상)

**목표**: "발명 설명 입력 → KIPO 양식 명세서 초안 자동 생성" 파이프라인 완성

- [ ] Ollama 설치 + 모델 후보 비교 (EXAONE 3.5 7.8B vs Qwen2.5 14B)
- [ ] 명세서 항목 스키마 정의 (명칭·기술분야·배경기술·해결과제·해결수단·효과·실시예·청구항)
- [ ] 섹션별 프롬프트 오케스트레이터 (순차 생성, 항목별 재생성)
- [ ] `file_samples/` PDF 파싱 → 입력 컨텍스트 활용 (few-shot 양식 참조)
- [ ] 웹 UI: 입력 폼 → 섹션별 결과 표시 → 재생성/편집
- [ ] Markdown / DOCX 내보내기
- **범위 제외(후순위)**: RAG, 파인튜닝, HWP, 사용자 인증

### Phase 2 — RAG (선행기술·유사특허 연동)

- KIPRIS 공개 특허 수집 → 임베딩(BGE-M3 등) → 벡터 DB
- 유사특허·선행문헌 검색 후 컨텍스트 주입 (신규성·청구항 작성 보조)
- 실제 명세서에서 유사 섹션 few-shot 자동 선별

### Phase 3 — 모델 특화 (파인튜닝, 선택)

- 명세서 corpus 기반 LoRA/QLoRA 파인튜닝 (문체·형식·용어 정합성↑)
- 평가 지표: 항목 완전성, 형식 준수율, 변리사 수정량 감소율

### Phase 4 — 운영·확장

- 품질 검증: 항목 누락 검사, 청구항 형식 검사, 도면-명세서 정합성
- HWP 출력, 도면 자동 인식·설명 생성
- 온프레미스 배포(Docker), 사용자 인증·이력 관리

## 5. 리스크 & 대응

| 리스크 | 대응 |
|---|---|
| 로컬 모델의 한국어·법률 문체 품질 | 모델 비교 + 항목 스키마·few-shot로 보정, Phase 3 파인튜닝, 필요시 32B급 |
| 환각(존재하지 않는 선행기술·인용) | MVP는 발명자 입력 범위 내 생성 유지, Phase 2 RAG로 근거 강화 |
| GPU 비용·자원 | MVP는 Ollama+중형으로 자원 최소화 후 필요시 확장 |
| 라이선스 (상용화 여부) | 모델 선정 시 상업 이용 조건 확인 (Qwen: Apache 2.0 유리) |

---

## 6. MVP 개발 착수 항목 (다음 단계)

- `backend/` : FastAPI 앱 (스키마 오케스트레이터 + Ollama 클라이언트 + PDF 파서)
- `frontend/`: 입력 폼 + 섹션별 결과 편집 UI
- 문서 파이프라인: PDF 파싱 + Markdown/DOCX 내보내기
- 시연 시나리오: 발명 설명 1건 입력 → 5개 이상 섹션이 채워진 명세서 초안 자동 생성
